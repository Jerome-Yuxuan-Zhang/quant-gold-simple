# 11 项目架构拆解（教材版）

本文档的目标不是“告诉你有哪些文件”，而是让你真正理解这个项目从数据到报告的整条链路。

## 1. 一句话理解这个仓库

这是一个面向黄金日频研究的可复现实验仓库。

它做的事情可以概括为：

1. 取数据
2. 统一成同一种时间序列格式
3. 生成特征
4. 定义预测目标
5. 做时间顺序切分
6. 训练基线模型
7. 生成评估与回测报告

如果你以后要做别的资产，比如原油、白银、股指、汇率，这套骨架仍然可以复用。

## 2. 架构总图

```text
configs/          -> 研究配置
data sources      -> 远程 API / 本地样本
normalizers       -> 统一字段和格式
features          -> 特征工程
targets           -> 预测目标定义
splitters         -> 时间顺序切分
models            -> 基线模型 / 模型比较 / 优化
backtest          -> walk-forward + 简化执行
reports           -> markdown/json/png 输出
```

## 3. 为什么要分层

很多初学者会把所有逻辑塞进一个 notebook。

这样短期很快，长期很危险：

- 改一处容易影响全部
- 难以单测
- 很难知道错误来自数据、特征还是模型
- 文档和代码很难同步

分层的价值在于：

- 每一层只负责一类问题
- 每一层都能单独验证
- 更容易定位“研究结论为什么变了”

## 4. 数据层

关键文件：

- `src/quant_gold/data/sources.py`
- `src/quant_gold/data/normalizers.py`
- `src/quant_gold/data/cache.py`
- `configs/datasets.json`

这一层回答的问题是：

- 去哪里拿数据
- 怎么缓存
- 不同来源的数据怎么统一

### 4.1 为什么要 normalizer

因为不同 API 的返回格式天生不一样。

如果你不先统一：

- 后面特征工程会写很多分支
- 同一个字段在不同数据源里名字不同
- 测试和重构都会非常痛苦

所以 normalizer 的使命是把它们统一成：

- `date`
- `value`
- `series_id`
- `source`
- `native_frequency`
- `unit`
- `last_updated`
- `description`

这一步是项目里很重要的“数据契约”。

## 5. 特征层

关键文件：

- `src/quant_gold/features/builders.py`

这一层做了几件核心事情：

- 以黄金价格为主时间轴
- 合并其它外生变量
- 对外生变量做保守滞后
- 计算收益率、动量、波动率、比值、z-score

### 5.1 为什么要对外生变量滞后

因为现实世界里，你未必能在同一时点无摩擦拿到所有宏观和风险代理变量。

如果你在特征构造里直接用“今天完整收盘后才知道的数据”去预测“今天”的结果，就会产生前视偏差。

所以这个项目默认对外生特征使用滞后，是一种偏保守的研究习惯。

## 6. 目标层

关键文件：

- `src/quant_gold/models/targets.py`

这层最容易被低估，但它其实决定了整个项目在研究上的可信度。

因为模型不是在预测“价格”，而是在预测你定义出来的标签。

如果标签定义有问题：

- 模型再复杂也没意义
- 回测很容易变成错觉

### 6.1 这里最重要的原则

预测目标和后续策略执行必须语义一致。

例如：

- 如果模型预测的是“未来 5 天累计收益方向”
- 那么策略就不能拿这个结果去只交易“明天一天”

本项目这次升级的一个重点，就是把多日目标下的持仓构造改成与持有期一致的 horizon-aligned position。

## 7. 评估层

关键文件：

- `src/quant_gold/models/baselines.py`
- `src/quant_gold/models/model_selection.py`
- `src/quant_gold/models/optimizer.py`
- `src/quant_gold/evaluation/metrics.py`

这一层负责回答：

- 基线模型表现如何
- 哪个模型更稳
- 有没有过拟合迹象

### 7.1 为什么不能用测试集选模型

测试集的意义是最终验收。

如果你在模型选择时已经看过测试集，再拿测试集宣布“模型有效”，这个结论就不再干净。

所以这次升级里，模型比较报告改成按验证集排序，而不是把测试集掺进“最佳模型”选择。

## 8. 回测层

关键文件：

- `src/quant_gold/backtest/walk_forward.py`
- `src/quant_gold/backtest/simulator.py`

这里解决两个不同问题：

### 8.1 Walk-forward

它回答的是：

“在不断向前滚动的现实时间轴里，模型是否还能工作？”

### 8.2 Simulator

它回答的是：

“即便模型能预测，这种预测变成仓位和收益后是否还有意义？”

请注意，这两个问题不是一回事。

## 9. 报告层

关键文件：

- `src/quant_gold/pipelines/v01_pipeline.py`
- `reports/generated/`

这个层的价值在于把研究结果固定成：

- markdown
- json
- png

这样你就能：

- 给别人看
- 做版本比较
- 回头审计自己

## 10. 你以后扩展时的顺序

如果你想把仓库继续做大，推荐顺序如下：

1. 先补更稳健的目标定义和切分策略
2. 再补更系统的特征工程
3. 再扩展模型池
4. 最后才做更复杂的回测和风险归因

不要反过来。

因为研究项目里，最容易骗人的往往不是“模型太差”，而是“流程看起来很完整，但定义不严谨”。
