# Contributing

感谢你愿意为 `quant-gold-simple` 做贡献。

这个项目既是一个量化研究起步仓库，也是一个偏教学型仓库，所以我们欢迎两类贡献：

- 工程质量提升：测试、CI、结构重构、异常处理、文档修复
- 研究质量提升：特征工程、目标定义、样本外验证、回测一致性

## 1. 先理解这个项目的边界

在提改动前，请先确认你的改动没有越过当前版本的研究边界：

- 这是日频研究项目，不是高频交易系统
- 这是研究基线，不是可直接部署的实盘系统
- 当前版本强调“可解释、可复现、可教学”
- 任何回测结果都必须被视为研究诊断，而不是收益承诺

## 2. 开发环境

建议使用 `uv`：

```bash
uv sync --extra dev
```

常用命令：

```bash
uv run ruff check .
uv run pytest
uv run quant-gold-v01 --mode sample
pre-commit install
```

## 3. 分支建议

推荐工作流：

1. 从 `main` 拉出功能分支
2. 小步提交，保持每次 commit 主题明确
3. 提交 PR 前先跑 lint、tests、sample pipeline
4. 如果修改了研究逻辑，必须同时解释“为什么这样更合理”

建议命名：

- `feat/add-x`
- `fix/repair-y`
- `docs/improve-z`
- `refactor/module-name`

## 4. 对研究代码的特殊要求

如果你修改以下模块，请特别小心：

- `targets.py`
- `walk_forward.py`
- `simulator.py`
- `model_selection.py`
- `optimizer.py`

原因很简单：这些文件最容易出现“代码能跑，但研究结论失真”的问题。

请始终自查：

- 有没有标签泄漏
- 有没有把测试集偷偷用于模型选择
- 预测目标和回测收益口径是否一致
- 滚动训练是否只使用过去信息
- 生成的报告是否准确描述了真实流程

## 5. 测试要求

至少满足：

- 新增功能要有测试
- 修复 bug 要补回归测试
- 不要只测“文件存在”
- 优先测试“研究语义是否正确”

好的测试例子：

- 多日目标下持仓构造是否与持有期一致
- API 返回限流或错误载荷时是否能明确报错
- 模型选择是否只依赖训练集和验证集

## 6. 文档要求

如果用户能感知到行为变化，就更新文档。

尤其是：

- README
- `docs/`
- 配置文件说明
- 报告解释文字

## 7. 不应提交的内容

不要提交以下内容：

- `.env`
- 真实 API key
- 本地缓存原始数据
- `data/interim/` 和 `data/processed/` 的派生产物
- `reports/generated/` 的临时报告
- `.venv/`、`__pycache__/` 等本地产物

## 8. PR 描述建议

一个高质量 PR，至少回答这几个问题：

1. 改了什么
2. 为什么改
3. 风险是什么
4. 怎么验证
5. 是否影响历史报告口径

## 9. 讨论风格

这个项目欢迎严谨的质疑。

如果你认为某个结果不可信，请优先指出：

- 数据问题
- 目标定义问题
- 评估问题
- 回测问题

这比“又加了一个更复杂模型”更有价值。
